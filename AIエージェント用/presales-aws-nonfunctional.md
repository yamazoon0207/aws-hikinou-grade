# AWSプリセールス - 非機能要件ヒアリングガイド

このガイドは、顧客のAWS非機能要件をヒアリングする際の補助ツールです。

## 使い方

チャットで以下のように依頼してください：
- 「非機能要件のヒアリングを開始したい」
- 「可用性要件について質問して」
- 「ヒアリング結果をまとめて」

## ヒアリングの流れ

1. **ビジネス要件の理解**
   - システムの目的と重要度
   - ユーザー数・トラフィック予測
   - ビジネスへの影響度

2. **非機能要件の確認**
   - 可用性（Availability）
   - パフォーマンス（Performance）
   - セキュリティ（Security）
   - コスト（Cost）
   - コンプライアンス（Compliance）
   - 運用性（Operability）

3. **AWSサービスマッピング**
   - 要件に応じたサービス提案
   - アーキテクチャパターンの提示

## 関連ファイル

詳細は各カテゴリ別のステアリングファイルを参照：
- `presales-availability.md` - 可用性要件
- `presales-performance.md` - パフォーマンス要件
- `presales-security.md` - セキュリティ要件
- `presales-cost.md` - コスト最適化
- `presales-compliance.md` - コンプライアンス
- `presales-operability.md` - 運用性

## ナレッジの追加方法

このファイルや各カテゴリファイルに、案件で得た知見を追加してください。
Gitで管理すれば、チーム全体で共有できます。

## 既存情報の確認

ヒアリングを開始する前に、以下の既存情報があるか確認してください。
これらがあると、ヒアリングがスムーズに進み、提案の精度が上がります。

### 1. 使用予定のAWSサービス
**質問:**
- 「すでに使用を想定されているAWSサービスはありますか？」
- 「社内で標準化されているAWSサービスはありますか？」
- 「使用禁止のAWSサービスはありますか？」

**確認ポイント:**
- 顧客が既に決めているサービス
- 社内標準やガイドラインで推奨されているサービス
- セキュリティポリシーで禁止されているサービス
- 既存システムで使用中のサービス（連携の可能性）

**例:**
- EC2、Lambda、S3、RDS、DynamoDB
- PrivateLink、Direct Connect
- CloudTrail、Config、GuardDuty
- 使用禁止: パブリックサブネットへの配置、など

### 2. 顧客が想定する構成図
**質問:**
- 「システム構成のイメージ図やラフスケッチはございますか？」
- 「参考にされているアーキテクチャパターンはありますか？」
- 「社内の他システムで採用している構成はありますか？」

**確認ポイント:**
- 顧客の頭の中にあるイメージ
- 参考にしているAWS公式のアーキテクチャ
- 社内の標準アーキテクチャ
- 他社事例で参考にしているもの

**提供形式:**
- PowerPoint、Visio、draw.io等の図
- 手書きのスケッチ
- AWS公式ドキュメントのURL
- 他社事例の資料

### 3. 現行環境の情報（移行案件の場合）

#### 3.1 現行システムのスペック
**質問:**
- 「現在のシステム構成図はございますか？」
- 「サーバーのスペック一覧（CPU、メモリ、ディスク）はありますか？」
- 「データベースのスペックと容量は？」

**確認ポイント:**
- サーバー台数とスペック
- ストレージ容量と使用率
- データベースの種類とバージョン
- ミドルウェアの種類とバージョン

**提供形式:**
- サーバー一覧表（Excel等）
- 構成管理台帳
- 運用手順書

#### 3.2 ネットワーク論理構成図
**質問:**
- 「ネットワーク構成図はございますか？」
- 「IPアドレス設計書はありますか？」
- 「ファイアウォールのルール一覧はありますか？」

**確認ポイント:**
- ネットワークセグメント構成
- IPアドレス体系
- ファイアウォールルール
- 外部システムとの接続方式
- VPN、専用線の有無

**提供形式:**
- ネットワーク構成図（Visio、PowerPoint等）
- IPアドレス設計書
- ファイアウォールルール一覧

#### 3.3 トラフィック情報
**質問:**
- 「現在のトラフィック量（リクエスト数、データ転送量）は把握されていますか？」
- 「ピーク時のトラフィックはどのくらいですか？」
- 「過去のトラフィック推移データはありますか？」

**確認ポイント:**
- 平常時のトラフィック
- ピーク時のトラフィック
- 季節変動の有無
- 過去の成長率

**提供形式:**
- 監視ツールのレポート
- アクセスログの分析結果
- トラフィックグラフ

#### 3.4 運用情報
**質問:**
- 「現在の運用手順書はございますか？」
- 「障害対応手順書はありますか？」
- 「バックアップ・リストア手順書はありますか？」

**確認ポイント:**
- 日次・週次・月次の運用作業
- 障害発生時の対応フロー
- バックアップ方式と頻度
- パッチ適用の方法とタイミング

**提供形式:**
- 運用手順書
- 障害対応手順書
- 運用カレンダー

### 4. セキュリティ・コンプライアンス情報

#### 4.1 セキュリティポリシー
**質問:**
- 「情報セキュリティポリシーはございますか？」
- 「クラウド利用に関するガイドラインはありますか？」
- 「セキュリティ基準書はありますか？」

**確認ポイント:**
- 暗号化の要件
- アクセス制御の要件
- ログ保管の要件
- 禁止事項

#### 4.2 コンプライアンス要件
**質問:**
- 「準拠すべき法規制やガイドラインのリストはありますか？」
- 「過去の監査レポートはありますか？」
- 「監査で指摘された事項はありますか？」

**確認ポイント:**
- 適用される法規制
- 業界標準
- 過去の監査結果
- 改善が必要な項目

### 5. 予算・スケジュール情報

#### 5.1 予算関連資料
**質問:**
- 「予算申請書や稟議書はございますか？」
- 「現行システムのコスト（TCO）は把握されていますか？」

**確認ポイント:**
- 承認済みの予算額
- 予算の内訳（初期費用、運用費用）
- 現行システムのコスト

#### 5.2 スケジュール
**質問:**
- 「プロジェクト計画書はございますか？」
- 「マイルストーンは決まっていますか？」

**確認ポイント:**
- 稼働開始日
- 主要マイルストーン
- 制約条件（契約満了日、EOSL等）

# AWS Presales Playbook

AWSプリセールス活動のためのナレッジベース・テンプレート集

## 📖 概要

このリポジトリは、AWS案件のプリセールス活動を効率化するためのナレッジベースです。
営業ヒアリング（BANTC）から非機能要件定義まで、実践的なテンプレートとガイドラインを提供します。

## 🎯 対象者

- AWSプリセールスエンジニア
- ソリューションアーキテクト
- 営業担当者
- プロジェクトマネージャー

## 📚 コンテンツ

### 1. 営業ヒアリング

#### [presales-bantc.md](presales-bantc.md)
AWS案件に特化したBANTCヒアリングガイド

- **B**udget（予算）: 初期費用と月額費用の確認
- **A**uthority（決裁権）: 決裁者と承認ルートの把握
- **N**eeds（必要性）: ビジネス目的と技術要件の深掘り
- **T**imeline（導入時期）: デッドラインと社内調整期間
- **C**ompetitor（競合）: 競合状況と差別化ポイント

**使い方:**
```
営業: 「BANTCヒアリングを開始したい」とKiroに依頼
→ Kiroが順番に質問をガイド
```

### 2. 非機能要件ヒアリング

#### [presales-aws-nonfunctional.md](presales-aws-nonfunctional.md)
非機能要件ヒアリングのメインガイド

- ヒアリングの流れ
- 既存情報の確認（使用予定サービス、構成図、現行環境）
- 各カテゴリへのリンク

**使い方:**

運用性要件のヒアリングガイド

- 障害対応（Runbook）

## 🚀 使い方

### 1. Kiroでの利用（推奨）

このリポジトリは、Kiro AIアシスタントのステアリングファイルとして設計されています。

#### セットアップ方法

**ステップ1: リポジトリをクローン**
```bash
cd ~
git clone <repository-url>
```

**ステップ2: Kiroのステアリングディレクトリに配置**

Kiroのステアリングファイルは以下のいずれかの場所に配置します：

1. **グローバル設定（全プロジェクトで利用）**
```bash
# ホームディレクトリの .kiro/steering/ にコピー
cp ~/aws-presales-playbook/presales-*.md ~/.kiro/steering/
```

2. **プロジェクト固有設定（特定プロジェクトのみ）**
```bash
# プロジェクトディレクトリに移動
cd /path/to/your/project

# プロジェクトの .kiro/steering/ にコピー
mkdir -p .kiro/steering
cp ~/aws-presales-playbook/presales-*.md .kiro/steering/
```

**ステップ3: Kiroを再起動**

ステアリングファイルを配置したら、Kiroを再起動するか、新しいチャットを開始してください。

**配置場所の違い:**
- `~/.kiro/steering/`: すべてのプロジェクトで常に利用可能
- `<project>/.kiro/steering/`: そのプロジェクトでのみ利用可能（プロジェクト固有の設定が優先）

#### 使用例
```
# 営業フェーズ
あなた: 「BANTCヒアリングを開始したい」
Kiro: 予算、決裁権、ニーズ、時期、競合を順番にヒアリング

# 技術ヒアリングフェーズ
あなた: 「非機能要件のヒアリングを開始したい」
Kiro: ビジネス要件 → 各非機能要件を深掘り

# 特定カテゴリのみ
あなた: 「セキュリティ要件について質問して」
Kiro: セキュリティ要件を詳細にヒアリング
```

## 📝 ナレッジの追加方法

実案件で得た知見を随時追加してください。

### 追加場所
各ファイルの最後に「## ナレッジ（随時追加）」セクションがあります。

### 追加例
```markdown
## ナレッジ（随時追加）

### 事例: 金融機関のFISC対応
**要件:** FISC安全対策基準への準拠
**対応:**
- 東京・大阪リージョンのマルチAZ構成
- CloudTrail、Configによる監査証跡
- KMSによる暗号化
**結果:** FISC準拠を実現し、社内監査を通過
```

### コミット
```bash
git add .
git commit -m "Add knowledge: 金融機関のFISC対応事例"
git push
```

## 📊 実践例

### 医療データ匿名加工プラットフォーム案件

このナレッジベースを使用した実案件の例：

**ヒアリング結果:**
- システム: 医療情報の匿名・仮名加工プラットフォーム
- データ量: 300GB/日（300医療機関）
- 予算: 初期1000万円、月額80万円
- RTO/RPO: 2-3日/1日
- コンプライアンス: 医療情報ガイドライン準拠

**成果物:**
- 非機能要件定義書
- アーキテクチャ設計書（2-VPC構成、PrivateLink）
- コスト見積書（3年TCO: 38.8M円）
- セキュリティ設計書（KMS 4キー設計）
- 運用設計書（Runbook含む）

詳細は `examples/` ディレクトリを参照。
