# AWS 非機能要件ヒアリングガイド

このガイドは、顧客のAWS非機能要件をヒアリングする際の補助ツールです。

## 使い方

チャットで以下のように依頼してください：
- 「非機能要件のヒアリングを開始したい」
- 「可用性要件について質問して」
- 「ヒアリング結果をまとめて」

## ヒアリングの流れ

1. **ビジネス要件の理解**
   - システムの目的と重要度
   - ユーザー数・トラフィック予測
   - ビジネスへの影響度

2. **非機能要件の確認**
   - 可用性（Availability）
   - パフォーマンス（Performance）
   - セキュリティ（Security）
   - コスト（Cost）
   - コンプライアンス（Compliance）
   - 運用性（Operability）

3. **AWSサービスマッピング**
   - 要件に応じたサービス提案
   - アーキテクチャパターンの提示

## 既存情報の確認

ヒアリングを開始する前に、以下の既存情報があるか確認してください。
これらがあると、ヒアリングがスムーズに進み、提案の精度が上がります。

### 1. 使用予定のAWSサービス
**質問:**
- 「すでに使用を想定されているAWSサービスはありますか？」
- 「社内で標準化されているAWSサービスはありますか？」
- 「使用禁止のAWSサービスはありますか？」

**確認ポイント:**
- 顧客が既に決めているサービス
- 社内標準やガイドラインで推奨されているサービス
- セキュリティポリシーで禁止されているサービス
- 既存システムで使用中のサービス（連携の可能性）

**例:**
- EC2、Lambda、S3、RDS、DynamoDB
- PrivateLink、Direct Connect
- CloudTrail、Config、GuardDuty
- 使用禁止: パブリックサブネットへの配置、など

### 2. 顧客が想定する構成図
**質問:**
- 「システム構成のイメージ図やラフスケッチはございますか？」
- 「参考にされているアーキテクチャパターンはありますか？」
- 「社内の他システムで採用している構成はありますか？」

**確認ポイント:**
- 顧客の頭の中にあるイメージ
- 参考にしているAWS公式のアーキテクチャ
- 社内の標準アーキテクチャ
- 他社事例で参考にしているもの

**提供形式:**
- PowerPoint、Visio、draw.io等の図
- 手書きのスケッチ
- AWS公式ドキュメントのURL
- 他社事例の資料

### 3. 現行環境の情報（移行案件の場合）

#### 3.1 現行システムのスペック
**質問:**
- 「現在のシステム構成図はございますか？」
- 「サーバーのスペック一覧（CPU、メモリ、ディスク）はありますか？」
- 「データベースのスペックと容量は？」

**確認ポイント:**
- サーバー台数とスペック
- ストレージ容量と使用率
- データベースの種類とバージョン
- ミドルウェアの種類とバージョン

**提供形式:**
- サーバー一覧表（Excel等）
- 構成管理台帳
- 運用手順書

#### 3.2 ネットワーク論理構成図
**質問:**
- 「ネットワーク構成図はございますか？」
- 「IPアドレス設計書はありますか？」
- 「ファイアウォールのルール一覧はありますか？」

**確認ポイント:**
- ネットワークセグメント構成
- IPアドレス体系
- ファイアウォールルール
- 外部システムとの接続方式
- VPN、専用線の有無

**提供形式:**
- ネットワーク構成図（Visio、PowerPoint等）
- IPアドレス設計書
- ファイアウォールルール一覧

#### 3.3 トラフィック情報
**質問:**
- 「現在のトラフィック量（リクエスト数、データ転送量）は把握されていますか？」
- 「ピーク時のトラフィックはどのくらいですか？」
- 「過去のトラフィック推移データはありますか？」

**確認ポイント:**
- 平常時のトラフィック
- ピーク時のトラフィック
- 季節変動の有無
- 過去の成長率

**提供形式:**
- 監視ツールのレポート
- アクセスログの分析結果
- トラフィックグラフ

#### 3.4 運用情報
**質問:**
- 「現在の運用手順書はございますか？」
- 「障害対応手順書はありますか？」
- 「バックアップ・リストア手順書はありますか？」

**確認ポイント:**
- 日次・週次・月次の運用作業
- 障害発生時の対応フロー
- バックアップ方式と頻度
- パッチ適用の方法とタイミング

**提供形式:**
- 運用手順書
- 障害対応手順書
- 運用カレンダー

### 4. セキュリティ・コンプライアンス情報

#### 4.1 セキュリティポリシー
**質問:**
- 「情報セキュリティポリシーはございますか？」
- 「クラウド利用に関するガイドラインはありますか？」
- 「セキュリティ基準書はありますか？」

**確認ポイント:**
- 暗号化の要件
- アクセス制御の要件
- ログ保管の要件
- 禁止事項

#### 4.2 コンプライアンス要件
**質問:**
- 「準拠すべき法規制やガイドラインのリストはありますか？」
- 「過去の監査レポートはありますか？」
- 「監査で指摘された事項はありますか？」

**確認ポイント:**
- 適用される法規制
- 業界標準
- 過去の監査結果
- 改善が必要な項目

### 5. 予算・スケジュール情報

#### 5.1 予算関連資料
**質問:**
- 「予算申請書や稟議書はございますか？」
- 「現行システムのコスト（TCO）は把握されていますか？」

**確認ポイント:**
- 承認済みの予算額
- 予算の内訳（初期費用、運用費用）
- 現行システムのコスト

#### 5.2 スケジュール
**質問:**
- 「プロジェクト計画書はございますか？」
- 「マイルストーンは決まっていますか？」

**確認ポイント:**
- 稼働開始日
- 主要マイルストーン
- 制約条件（契約満了日、EOSL等）

## ヒアリング前の準備チェックリスト

ヒアリングを開始する前に、以下をチェックしてください：

- [ ] 使用予定のAWSサービスリストを入手
- [ ] 顧客想定の構成図を入手（あれば）
- [ ] 現行システムのスペック一覧を入手（移行案件の場合）
- [ ] ネットワーク構成図を入手（移行案件の場合）
- [ ] トラフィック情報を入手（移行案件の場合）
- [ ] セキュリティポリシーを入手
- [ ] コンプライアンス要件リストを入手
- [ ] 予算情報を入手
- [ ] プロジェクトスケジュールを入手

これらの情報が揃っていない場合でも、ヒアリングを通じて確認していきます。
